### 1.2 各维度详细评估

#### 产品愿景与定位（9/10）

QuitMate 的三阶段模型（认知觉醒 → 行动改变 → 维持防复发）直接对齐了跨理论模型（TTM），这是整个戒烟戒酒领域中最被忽视的环节——**前意向阶段（Precontemplation）的意识唤醒**。市面上几乎所有产品都假设用户"已经想戒了"，而 QuitMate 从"Reality Check"（现实检验）开始，通过数据可视化让用户自己看到吸烟/饮酒的真实代价。这是一个极具洞察力的产品定位。

**扣分点**：Reality Check 的交互设计可以更加沉浸式，目前偏重数字输入，缺少情感冲击力。

#### 架构设计（8/10）

采用了 Clean Architecture 分层（`presentation → domain → data → core`），这在 Flutter 社区中属于高质量实践。关键设计亮点包括：

- **领域驱动**：`UserStage` 枚举直接映射 TTM 模型的 5 个阶段，`GameProfile` 的等级体系设计精细
- **关注点分离**：AI Coach、Pattern Analyzer、Daily Task Generator 各自独立，可独立测试和替换
- **依赖注入**：使用 Riverpod 实现了 Provider 层的 DI
- **安全层**：AES-256 加密敏感数据，使用 flutter_secure_storage 保护密钥

**扣分点**：
- 部分 Screen 文件过大（如 `dashboard_screen.dart` 45KB、`skills_lab_screen.dart` 51KB），缺乏子组件拆分
- `ai_analyst.dart`（50KB）过于庞大，分析逻辑和数据展示耦合
- Repository 层直接操作 SQL，缺少数据访问抽象层

#### 功能完整度（8/10）

在 v1.8.0 版本中，QuitMate 已覆盖了三阶段模型的完整功能链：

| 渴望冲浪 / SOS 呼吸法 | ✅ 已实现 | 85% |
| 技能实验室（30+ CBT 练习） | ✅ 已实现 | 80% |
| 每日任务系统 | ✅ 已实现 | 75% |
| 游戏化（等级/XP/徽章） | ✅ 已实现 | 85% |
| 虚拟伙伴（小明） | ✅ 已实现 | 70% |
| 数据分析报告 | ✅ 已实现 | 80% |

**扣分点**：LLM 集成尚不完整，虚拟伙伴的对话能力偏模板化，缺少真正个性化的上下文记忆。

#### 代码质量（6/10）

**亮点**：
- 统一的 Design System（AppTheme），灵感来自 Headspace/Calm/Noom，规范度高
- 数据库版本管理做得不错，从 v1 迭代到 v6 的 migration 逻辑清晰
- Entity 设计合理，`copyWith` 模式使用一致

**问题**：
- 几乎没有单元测试（`test/widget_test.dart` 仅 211 字节，是默认模板文件）
- 大量硬编码的中文字符串散落在 Dart 文件中，没有使用 `intl` 的 `message lookup` 机制
- 部分 Screen 文件职责过重（5000+ 行），违反单一职责原则
- `ai_coach_engine.dart` 中有大量 `_randomFrom([...])` 模式重复

#### 测试覆盖（2/10）

这是目前最大的短板。整个项目只有 Flutter 默认生成的 `widget_test.dart` 模板文件，没有任何实质性的单元测试、组件测试或集成测试。对于一个健康/医疗类应用，这构成了严重的质量风险。

**亟需补充的测试**：
- `PatternAnalyzer` 的统计计算（边界条件、空数据、极端值）
- `AiCoachEngine` 的对话分类和上下文逻辑
- `GameProfile` 的 XP 计算和升级逻辑
- `EncryptionService` 的加解密正确性
- Repository 层的 CRUD 操作
- 数据库 migration 的向前/向后兼容性

#### 隐私与安全（9/10）

**亮点**：
- 纯离线架构，所有数据存储在本地 SQLite
- AES-256-CBC 加密敏感用户偏好数据
- 使用 `flutter_secure_storage` 保护密钥
- README 明确声明"不收集任何个人信息"
- `key.properties` 存在本地，未暴露到代码仓库（通过 .gitignore 管理）

**注意**：
- `key.properties` 文件存在于仓库中（虽是本地签名用），需确认未包含敏感信息
- LLM 集成（`http` 依赖）会打破纯离线承诺，需要在 UI 上给用户明确提示

#### UI/UX 设计（8/10）

**亮点**：
- 灵感来自 Headspace/Calm 的 Nature-infused 设计语言，在戒烟戒酒类应用中独树一帜
- 完整的语义色彩系统（AppColors），支持亮色/暗色模式
- 自定义 SVG 徽章资源
- 底部导航（Shell）结构清晰，信息架构合理

**扣分点**：
- 过大的 Screen 文件意味着 UI 组件没有充分拆分，复用性受限
- 缺少无障碍（Accessibility）支持标注

### 1.3 核心优势

1. **三阶段模型的完整覆盖**：从"不想戒"到"戒了不复发"的全链路干预，这是市面上独一无二的定位
2. **纯离线 + 隐私优先**：在数据隐私日益敏感的时代，这是一个强大的差异化卖点
3. **科学方法论支撑**：FTND/AUDIT 标准量表、CBT 技能库、TTM 模型对齐，学术基础扎实
4. **游戏化体系设计精良**：等级系统、XP 奖励、徽章收集的设计逻辑完整且可扩展
5. **本地 AI 分析引擎**：Pattern Analyzer 在完全离线的情况下提供个性化数据分析，技术含量高

## 二、产品定位与差异化策略

### 目标用户画像

**核心用户**：25-45 岁的中国男性，有吸烟或饮酒习惯，但处于"知道不好但还没准备好戒"的犹豫状态。他们可能曾经尝试过戒烟/戒酒但失败了，对"被教育"感到厌倦。

**次要用户**：家人朋友希望帮助亲人戒烟戒酒的"关心者"群体。

### 差异化三角

```
                    隐私安全
                   ／     ＼
                 ／         ＼
               ／  QuitMate   ＼
             ／               ＼
    科学方法论 ──────── 三阶段全链路
```

1. **vs 传统打卡类**（戒烟军团等）：我们有完整的干预体系，不是简单的打卡计数器
2. **vs 国外 CBT 类**（Kwit、Reframe）：我们覆盖"意识唤醒"阶段，不要求用户已有意愿
3. **vs 医疗类**（戒烟门诊）：我们随时陪伴在用户手机里，零门槛零成本

## 三、技术架构演进路线

### 当前架构（v1.x）

```
┌─────────────────────────────────┐
│         Presentation           │  ← GoRouter + Riverpod
│  (Screens / Widgets / Theme)   │
├─────────────────────────────────┤
│            Domain              │  ← Entities / UseCases
│ (User / GameProfile / Craving) │
├─────────────────────────────────┤
│             Data               │  ← SQLite / Repository
│  (AppDatabase / Repositories)  │
├─────────────────────────────────┤
│             Core               │  ← AI Coach / Encryption / DI
│ (AiCoachEngine / Services)    │
└─────────────────────────────────┘
```

### 目标架构（v3.0）

```
┌─────────────────────────────────────────┐
│              Presentation                │
│  ┌─────────┐ ┌─────────┐ ┌──────────┐ │
│  │ Screens │ │ Widgets │ │ Platform  │ │  ← iOS + Android + Widget
│  │(拆分后) │ │(可复用) │ │Adapters  │ │
│  └─────────┘ └─────────┘ └──────────┘ │
├─────────────────────────────────────────┤
│              Domain (不变)               │  ← 纯 Dart，零依赖
├─────────────────────────────────────────┤
│              Data                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │ SQLite   │ │ HealthKit│ │(可选)云端│ │  ← 抽象接口，可替换实现
│  │(Drift ORM)│ │ 适配器   │ │  同步    │ │
│  └──────────┘ └──────────┘ └──────────┘ │
├─────────────────────────────────────────┤
│              Core                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │规则引擎   │ │ ML推理   │ │ 端侧LLM  │ │  ← 100% 离线运行
│  │AiCoach   │ │PatternAI │ │(可选集成)│ │
│  └──────────┘ └──────────┘ └──────────┘ │
└─────────────────────────────────────────┘
```

## 四、产品功能演进路线

### Phase 1：夯实基础（v2.0）

**目标**：在不联网的前提下，提升 App 的日常打开率、干预有效性和工程质量。

**预计周期**：4-6 周

#### 1.1 工程质量提升

- [ ] **单元测试覆盖核心业务逻辑 ≥ 60%**
  - `PatternAnalyzer` 统计分析测试（边界值、空数据、极端数据）
  - `AiCoachEngine` 对话分类和上下文匹配测试
  - `GameProfile` XP 计算、升级、连胜逻辑测试
  - `DailyTaskGenerator` 任务生成逻辑测试
  - `EncryptionService` 加解密正确性测试
  - `UserRepository` / `CravingRepository` CRUD 测试
  - 数据库 migration 兼容性测试
- [ ] **Screen 组件拆分**
  - `dashboard_screen.dart`（45KB）→ 拆分为 `HealthMilestoneCard`、`StreakWidget`、`DailyOverview`、`QuickActions` 等子组件
  - `skills_lab_screen.dart`（51KB）→ 拆分为 `ExerciseCard`、`CategoryFilter`、`ExerciseDetail` 等
  - `analysis_report_screen.dart`（39KB）→ 拆分为 `InsightCard`、`PatternChart`、`TrendGraph` 等
  - `ai_analyst.dart`（50KB）→ 按分析维度拆分为 `TimePatternAnalyzer`、`MoodAnalyzer`、`TrendAnalyzer` 等
- [ ] **国际化重构**
  - 引入 Flutter ARB 文件机制
  - 所有硬编码字符串迁移到 `lib/l10n/`
  - 首批支持：中文简体、英文
- [ ] **CI/CD 修复与增强**
  - 修复 `ci.yml` 中 `branches: ain` 语法错误
  - 添加 `test/` 目录下所有测试到 CI 流水线
  - 添加 test coverage 阈值门禁（≥ 60%）
  - 添加 release APK 自动签名和上传

#### 1.2 产品功能增强

- [ ] **健康平台数据集成（HealthKit / Google Fit）**
  - 获取心率和睡眠数据
  - 当检测到高压力/低睡眠时主动推送本地干预通知
  - 将戒烟/戒酒进度同步到健康 App（可选）
- [ ] **Reality Check 沉浸式升级**
  - 加入身体年龄对比动画（"你的肺年龄是 X 岁"）
  - 加入金钱时间线可视化（"这些年你花在烟/酒上的钱可以买..."）
  - 加入社交对比数据（匿名统计："80% 的人在你的年龄段..."）
- [ ] **游戏化增强**
  - 引入"虚拟植物/宠物"养成机制（利用损失厌恶心理）
  - 金钱节省可视化升级（从文字变为动画积攒效果）
  - 每周生成个人成就海报（可分享，不含个人信息）
- [ ] **渴望冲浪（Urge Surfing）引导工具升级**
  - 3-5 分钟可视化呼吸引导界面
  - 与渴望日志联动，记录每次冲浪结果
  - 渐进解锁不同冲浪场景（海滩、森林、星空）

#### 1.3 技术债务清理

- [ ] 统一 Repository 层数据访问模式，引入接口抽象
- [ ] `AiCoachEngine` 重构：将 `_randomFrom` 模式改为配置驱动的响应模板系统
- [ ] 数据库 schema 审查与优化，为 HealthKit 数据预留表结构
- [ ] 升级关键依赖到最新稳定版

### Phase 2：智能升级（v3.0）

**目标**：引入端侧智能能力，提升个性化干预精度；完成 iOS 适配，扩大用户覆盖。

**预计周期**：6-8 周

#### 2.1 端侧机器学习

- [ ] **渴望预测模型**
  - 基于用户历史数据训练时间序列模型
  - 在高危时段主动推送预防性干预
  - 使用 TensorFlow Lite 或 ONNX Runtime 在端侧运行
- [ ] **复发风险评估**
  - 综合连胜天数、近期渴望频率、情绪波动等因素
  - 生成每日"风险分数"，在 Dashboard 上展示
  - 高风险时触发 SOS 主动关怀
- [ ] **个性化干预优化**
  - A/B 测试框架（本地）用于对比不同干预话术效果
  - 基于用户反馈自动调整干预策略权重

#### 2.2 iOS 平台完整支持

- [ ] iOS 原生 Widget（iOS 14+ WidgetKit）
- [ ] iOS HealthKit 集成
- [ ] iOS 暗色模式适配
- [ ] iOS 推送通知（本地）
- [ ] App Store 上架准备（截图、描述、隐私政策）

#### 2.3 社区功能（纯离线匿名）

- [ ] **复发宽容机制（Slide, not a fall）**
  - 意外破戒记录功能，分析触发原因
  - 进度重置但不抹杀历史
  - 引导用户继续而非自暴自弃
- [ ] **离线社区内容**
  - 预置高质量的成功故事和经验分享（定期更新内容包）
  - 用户可匿名提交自己的故事（经审核后进入下一版本内容包）

#### 2.4 LLM 策略明确化

- [ ] 定义清晰的 LLM 使用边界：
  - **默认模式**：100% 离线，使用规则引擎 AI Coach
  - **可选增强模式**：用户主动开启后，连接端侧 LLM（如 Llama.cpp）或远程 API
  - 所有网络请求前必须获得用户明确授权
  - LLM 响应仅作为建议，不替代规则引擎的核心安全判断

### Phase 3：生态扩展（v4.0）

**目标**：从个人工具升级为可被专业场景使用的干预平台。

**预计周期**：8-12 周

#### 3.1 可选云端同步（零知识架构）

- [ ] 用户自主选择是否开启云同步
- [ ] 端到端加密（E2EE），服务器无法解密用户数据
- [ ] 支持设备迁移和数据备份恢复
- [ ] 多设备同步（手机 + 平板）

#### 3.2 医疗/机构接口

## 七、产品设计红线

> 基于对市场竞品的深度分析和行为心理学原理，以下设计决策必须坚决避免：

1. **禁止恐吓式科普**：过度展示烂肺、黑肝图片会引起心理防御机制，导致用户逃避。应多做"正向收益"展示。
2. **禁止在用户脆弱时展示广告**：烟瘾/酒瘾发作时的焦虑状态下推送广告，会瞬间摧毁信任并可能引发报复性复吸。
3. **禁止强制社交绑定**：戒烟戒酒是极度私密的事情，强制绑定微信/朋友圈将导致 80% 用户在注册第一步流失。
4. **禁止功能蔓延**：不要同时做戒烟、戒酒、戒色、减肥、戒赌。先死磕烟/酒，把三阶段模型跑通做深。
5. **禁止降低隐私承诺**：任何涉及网络请求的功能必须是用户"主动选择开启"的 opt-in 模式，且必须在 UI 上有明确的隐私提示。
6. **禁止"全或无"的复发处理**：用户破戒一次就清零所有数据、让用户从零开始，这种"惩罚式设计"会导致用户自暴自弃彻底卸载 App。
